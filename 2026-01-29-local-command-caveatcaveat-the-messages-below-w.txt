❯ swipe 까지 만들어야함? 원래 그런가 안느려?                                    
                                                                                
⏺ 좋은 질문입니다. Swipe 기록을 저장하는 이유는:                                
                                                                                
  1. 중복 방지 — 같은 사람을 두 번 swipe하지 않도록                             
  2. 매칭 판단 — 상대가 나를 LIKE 했는지 확인해야 Match를 만들 수 있음          
  3. 추천 필터링 — 이미 swipe한 사람은 추천 목록에서 제외                       
                                                                                
  Swipe 테이블 없이는 매칭 자체가 불가능합니다. "A가 B를 좋아했다"는 기록이     
  있어야 B가 A를 좋아했을 때 매칭을 만들 수 있으니까요.                         
                                                                                
  성능 걱정은:                                                                  
  - 단순한 INSERT + SELECT 쿼리라 느리지 않습니다                               
  - (from_user_id, to_user_id)에 유니크 인덱스 걸면 충분히 빠릅니다             
                                                                                
  결론: Swipe 테이블은 필수입니다. 만들어볼까요?                                
                                                                                
❯ 그럼 테이블이 앞으로 채팅방 만들떄까지 뭐가 더 필요해? swipe랑                
                                                                                
⏺ 현재부터 채팅방까지 필요한 테이블:                                            
  ┌─────────────┬────────────────────────┬─────────┐                            
  │   테이블    │          용도          │   DB    │                            
  ├─────────────┼────────────────────────┼─────────┤                            
  │ Swipe       │ Like/Pass 기록         │ MySQL   │                            
  ├─────────────┼────────────────────────┼─────────┤                            
  │ Match       │ 양쪽 LIKE 시 매칭 기록 │ MySQL   │                            
  ├─────────────┼────────────────────────┼─────────┤                            
  │ ChatRoom    │ 매칭된 쌍의 채팅방     │ MongoDB │                            
  ├─────────────┼────────────────────────┼─────────┤                            
  │ ChatMessage │ 채팅 메시지            │ MongoDB │                            
  └─────────────┴────────────────────────┴─────────┘                            
  이미 있는 것: User, Hobby, UserHobby                                          
                                                                                
  흐름은 이렇습니다:                                                            
                                                                                
  Swipe(LIKE) → 상대도 LIKE? → Match 생성 → ChatRoom 자동 생성 → ChatMessage    
  주고받기                                                                      
                                                                                
  채팅은 MongoDB(이미 설정되어 있음), 나머지는 MySQL입니다. 이 구조로 Swipe부터 
  순서대로 진행할까요?                                                          
                                                                                
❯ match도 필요해?                                                               
                                                                                
⏺ Match 없이도 가능하긴 합니다. Swipe 테이블만으로 "서로 LIKE" 여부를 쿼리할 수 
  있거든요.                                                                     
                                                                                
  Match 테이블 없이:                                                            
  -- A와 B가 서로 LIKE인지 확인                                                 
  SELECT * FROM swipe                                                           
  WHERE from_user=A AND to_user=B AND type=LIKE                                 
  AND EXISTS (                                                                  
    SELECT 1 FROM swipe                                                         
    WHERE from_user=B AND to_user=A AND type=LIKE                               
  )                                                                             
                                                                                
  근데 Match 테이블을 따로 두는 이유:                                           
  - 매칭 목록 조회가 매번 JOIN 없이 바로 됨                                     
  - 매칭 시간, 상태(활성/차단/해제) 관리 가능                                   
  - 채팅방과 1:1 연결이 깔끔함                                                  
                                                                                
  규모가 작으면 없어도 되고, 있으면 나중에 편합니다. 어떻게 하시겠어요?         
                                                                                
❯ @manytoone 위에 관련되서 설명좀                                               
                                                                                
⏺ @ManyToOne은 DB 관계를 표현하는 JPA 어노테이션입니다.                         
                                                                                
  Swipe 예시로 설명하면:                                                        
                                                                                
  @Entity                                                                       
  public class Swipe {                                                          
      @Id @GeneratedValue                                                       
      private Long id;                                                          
                                                                                
      @ManyToOne(fetch = FetchType.LAZY)                                        
      private User fromUser;  // swipe한 사람                                   
                                                                                
      @ManyToOne(fetch = FetchType.LAZY)                                        
      private User toUser;    // swipe 당한 사람                                
  }                                                                             
                                                                                
  "Many To One" = 여러 개 → 하나:                                               
  - Swipe 여러 개가 User 하나를 가리킴                                          
  - 한 유저가 여러 번 swipe 할 수 있으니까 → Many(Swipe) To One(User)           
                                                                                
  DB에선 이렇게 됨:                                                             
                                                                                
  swipe 테이블                                                                  
  | id | from_user_id (FK) | to_user_id (FK) | type |                           
  |----|-------------------|-----------------|------|                           
  | 1  | 3                 | 7               | LIKE |                           
  | 2  | 3                 | 5               | PASS |                           
  | 3  | 7                 | 3               | LIKE |                           
                                                                                
  from_user_id, to_user_id는 자동으로 FK(외래키) 컬럼이 생깁니다.               
                                                                                
  fetch = FetchType.LAZY는:                                                     
  - Swipe를 조회할 때 User 정보를 바로 안 가져옴                                
  - swipe.getFromUser() 호출할 때 그때 쿼리 실행                                
  - 불필요한 JOIN을 방지해서 성능에 좋음                                        
                                                                                
  이미 프로젝트에 있는 UserHobby.java도 같은 패턴입니다 — User와 Hobby에 각각   
  @ManyToOne이 걸려있죠.                                                        
                                                                                
❯ joincolumn 이런건 필요없나?                                                   
                                                                                
⏺ @JoinColumn은 FK 컬럼명을 직접 지정할 때 씁니다.                              
                                                                                
  없으면:                                                                       
  @ManyToOne                                                                    
  private User fromUser;                                                        
  // → 컬럼명 자동생성: from_user_id                                            
                                                                                
  있으면:                                                                       
  @ManyToOne                                                                    
  @JoinColumn(name = "from_id")                                                 
  // → 컬럼명: from_id (직접 지정)                                              
                                                                                
  자동생성 규칙이 필드명_id라서, fromUser → from_user_id로 알아서 만들어줍니다. 
                                                                                
  @JoinColumn을 쓰는 경우:                                                      
  - 컬럼명을 직접 정하고 싶을 때                                                
  - nullable = false 같은 제약조건 추가할 때                                    
  - 레거시 DB에 맞춰야 할 때                                                    
                                                                                
  지금 프로젝트의 UserHobby.java도 @JoinColumn 없이 @ManyToOne만 쓰고 있어서,   
  같은 스타일로 가면 됩니다. 안 써도 문제없습니다.                              
                                                                                
❯ @OneToMany(mappedBy = "user")                                                 
  private List<Swipe> swipes = new ArrayList<>();                               
                                                                                
⏺ 이건 User 엔티티에 넣는 건데, 꼭 필요하지는 않습니다.                         
                                                                                
  @OneToMany가 하는 일:                                                         
  - User 객체에서 user.getSwipes()로 바로 접근 가능하게 해줌                    
  - DB에는 아무 변화 없음 (FK는 Swipe 테이블에 있으니까)                        
                                                                                
  mappedBy = "user"의 의미:                                                     
  - "Swipe 엔티티의 user 필드가 이 관계의 주인이다"                             
  - 즉, FK 관리는 Swipe 쪽에서 한다는 뜻                                        
                                                                                
  넣으면: user.getSwipes()로 편하게 조회 가능                                   
  안 넣으면: swipeRepository.findByFromUser(user)로 조회하면 됨                 
                                                                                
  현재 프로젝트에서 User에 @OneToMany로 userHobbies가 이미 있는데, 그건 프로필  
  업데이트 시 cascade 삭제가 필요해서 넣은 겁니다.                              
                                                                                
  Swipe는 User 삭제 시 같이 지울 게 아니라면 안 넣어도 됩니다. Repository에서   
  조회하는 게 더 깔끔해요.                                                      
                                                                                
❯ 잘 이해가 안되는데 어떨때 쓰고 어떨떄 안쓰는지 그리고 user 삭제 될떄          
  없어져야하는거 아닌가? 다른건가?                                              
                                                                                
⏺ 언제 쓰는가:                                                                  
                                                                                
  @OneToMany는 부모 엔티티에서 자식을 직접 관리해야 할 때 씁니다.               
                                                                                
  // User 엔티티에 이미 있는 예시                                               
  @OneToMany(mappedBy = "user", cascade = CascadeType.ALL, orphanRemoval = true)
  private List<UserHobby> userHobbies = new ArrayList<>();                      
                                                                                
  이게 있으면:                                                                  
  - user.getUserHobbies().clear() → 취미 전부 삭제됨                            
  - user.getUserHobbies().add(new UserHobby(...)) → 취미 추가됨                 
  - User 삭제 → UserHobby도 자동 삭제 (cascade = ALL)                           
                                                                                
  프로필 업데이트 시 기존 취미를 지우고 새로 넣어야 하니까 이 방식이 편한       
  거예요.                                                                       
                                                                                
  Swipe는 왜 다른가:                                                            
                                                                                
  Swipe는 User에서 직접 관리할 일이 없습니다.                                   
  - 추가: swipeRepository.save(new Swipe(...))                                  
  - 조회: swipeRepository.findByFromUser(user)                                  
  - User가 Swipe를 add/remove 할 필요 없음                                      
                                                                                
  User 삭제 시 Swipe 삭제는?                                                    
                                                                                
  맞습니다, 삭제돼야 합니다. 근데 두 가지 방법이 있어요:                        
  ┌────────────────────────┬─────────────────┐                                  
  │          방법          │      코드       │                                  
  ├────────────────────────┼─────────────────┤                                  
  │ @OneToMany + cascade   │ JPA가 자동 삭제 │                                  
  ├────────────────────────┼─────────────────┤                                  
  │ DB의 ON DELETE CASCADE │ DB가 자동 삭제  │                                  
  └────────────────────────┴─────────────────┘                                  
  JPA cascade는 User 객체를 통해 관리할 때 쓰고, DB cascade는 그냥 삭제만       
  연동하면 될 때 씁니다.                                                        
                                                                                
  정리:                                                                         
  ┌───────────────────────────┬─────────────────┬───────────────────┐           
  │                           │    UserHobby    │       Swipe       │           
  ├───────────────────────────┼─────────────────┼───────────────────┤           
  │ User에서 add/remove 하나? │ O (프로필 수정) │ X                 │           
  ├───────────────────────────┼─────────────────┼───────────────────┤           
  │ @OneToMany 필요?          │ O               │ X                 │           
  ├───────────────────────────┼─────────────────┼───────────────────┤           
  │ 삭제 연동                 │ JPA cascade     │ DB cascade로 충분 │           
  └───────────────────────────┴─────────────────┴───────────────────┘           
                                                                                
❯ 그럼 굳이 들고 있을 필요가 없네? 그냥 swipe만 가지고 있으면 되겠네

  - 내가 swipe한 사람들 목록?                                                   
  - 나를 좋아한 사람들 목록?                                                    
  - 아니면 다른 거?                                                             
                                                                                
  어떤 기능을 만들려는 건지 알려주시면 정확하게 알려드릴게요.                   
                                                                                
❯ 아니 swipe가 뭘 위해 있는건데? 이게 들억마ㅕㄴ 추천 사람 떠서 거기서          
  선택해야하는데 추천 사람이 swipe가 아닌가?                                    
                                                                                
⏺ 아, 이해했습니다. 헷갈린 부분을 정리하면:                                     
                                                                                
  추천 목록 ≠ Swipe                                                             
  ┌───────────┬───────────────────────────────────────────────────────────┐     
  │   기능    │                           설명                            │     
  ├───────────┼───────────────────────────────────────────────────────────┤     
  │ 추천 목록 │ 아직 swipe 안 한 사람들 중 나에게 맞는 사람을 보여주는 것 │     
  ├───────────┼───────────────────────────────────────────────────────────┤     
  │ Swipe     │ 추천된 사람을 보고 LIKE 또는 PASS를 누르는 행위의 기록    │     
  └───────────┴───────────────────────────────────────────────────────────┘     
  흐름:                                                                         
  추천 목록 조회 → 사람 한 명 보여줌 → LIKE/PASS 누름 → Swipe 저장 → 다음 사람  
                                                                                
  그래서 필요한 API는:                                                          
                                                                                
  1. GET /api/recommendations — 추천 사람 목록 가져오기 (이미 swipe한 사람 제외)
  2. POST /api/swipes — LIKE 또는 PASS 저장                                     
                                                                                
  "getSwipes"가 아니라 추천 목록 조회를 먼저 만들어야 하는 거였네요. 추천       
  로직부터 갈까요?                                                              
                                                                                
❯ 아니면 그냥 순수하게 그냥 눌렀을때 그 결과를 저장하는 거였어?                 
                                                                                
⏺ 네, 맞습니다. Swipe는 그냥 눌렀을 때 결과를 저장하는 것입니다.                
                                                                                
  화면에 추천된 사람 뜸 → LIKE 누름 → Swipe 테이블에 저장                       
                                                                                
  그게 전부입니다. Swipe 테이블은 단순한 기록 저장소예요.                       
                                                                                
  그리고 그 기록이 있어야:                                                      
  - 상대도 나를 LIKE 했는지 확인 → Match 생성                                   
  - 이미 swipe한 사람은 추천에서 제외                                           
                                                                                
  순서대로 하면:                                                                
  1. Swipe 저장 API ← 지금 이거                                                 
  2. 추천 목록 API                                                              
  3. Match 로직                                                                 
                                                                                
  Swipe 저장부터 만들어보세요.                                                  
                                                                                
❯ swipe 저장

  public void swipe(Long fromUserId, Long toUserId, SwipeType type) {

⏺ 몇 가지 수정할 점:                                                            
                                                                                
  1. swipeRepository. 지워야 함 — 미완성 코드가 남아있어요.                     
                                                                                
  2. SwipeRequest에서 User 객체가 아니라 ID를 받아야 합니다.                    
  클라이언트가 보내는 건 { "toUserId": 5, "type": "LIKE" } 이니까:

  private SwipeType type;

  돼요 — 다른 사람인 척 swipe할 수 있으니까.                                    
                                                                                
  3. 중복 체크가 빠져있음:

❯ 상대방 정보는 어디서 받아옴?                                                  
                                                                                
⏺ 클라이언트(프론트)에서 보내줍니다.                                            
                                                                                
  추천 목록 화면에서 사람이 뜨잖아요 → 그 사람의 ID를 알고 있으니까 → LIKE      
  누르면 그 ID를 서버로 보내는 겁니다.                                          
                                                                                
  프론트 흐름:                                                                  
  1. GET /api/recommendations → [{ id: 5, nickname: "철수" }, { id: 7, nickname:
   "영희" }]                                                                    
  2. 철수 카드에서 LIKE 누름                                                    
  3. POST /api/swipes → { "toUserId": 5, "type": "LIKE" }                       
                                                                                
  서버는 toUserId만 받으면 됩니다. 그래서 SwipeRequest는:                       
                                                                                
  private Long toUserId;                                                        
  private SwipeType type;                                                       
                                                                                
  이것만 있으면 돼요.
